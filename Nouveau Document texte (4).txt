Je propose de renommer la prochaine version :

Cortex-3 — Recursive Verified Cognitive Compiler

Pas seulement un LLM compressé.

Pas seulement un BitNet amélioré.

Pas seulement un système avec vérificateur.

Mais :

un modèle qui utilise le raisonnement lent pour découvrir de nouvelles compétences, puis compile ces compétences en circuits ternaires rapides, vérifiés, réutilisables, et de moins en moins coûteux.

C’est là qu’apparaît le vrai changement de paradigme :

ancien paradigme :
entraîner un modèle → l’utiliser

nouveau paradigme :
faire résoudre lentement → vérifier → compiler → accélérer → réutiliser → améliorer le prochain cycle

Le cœur du projet reste celui que nous avons construit : poids ternaires {-1,0,+1}, compression d’activations, KV latent, MTP/FSP, raisonnement latent, experts ternaires, vérificateur, regrowth et sleep phase. Le plan initial décrit déjà Cortex comme un modèle qui apprend à devenir une forme minimale et vérifiable de lui-même, au lieu d’être simplement compressé après entraînement. Et le plan Cortex-2 a déjà identifié le vrai changement : passer d’une architecture à un compilateur cognitif vérifié, avec Bit Ledger, Skill Ledger, Causal Ledger et Uncertainty Ledger.

Mais maintenant, on va aller plus loin.

1. Le prochain goulot : le vérificateur de compétences rares

Le prochain goulot n’est pas BitNet.

Ce n’est pas MTP.

Ce n’est pas le KV cache.

Ce n’est même pas le raisonnement latent.

Le prochain goulot est :

Comment prouver qu’une compression n’a pas détruit une compétence rare, fragile, ou difficile à observer ?

C’est le point critique.

Parce qu’un modèle peut conserver :

la fluidité,
le style,
la réponse moyenne,
la perplexité,
les benchmarks visibles,
les tâches fréquentes,

tout en perdant silencieusement :

la précision mathématique rare,
la robustesse aux reformulations,
le raisonnement long,
le suivi exact de variables,
la calibration,
la mémoire de détails,
la capacité de dire “je ne sais pas”,
la résistance aux pièges.

Ton plan l’avait déjà formulé : le vérificateur doit tester les compétences, pas simplement la loss moyenne, et la compression ne doit pas être acceptée si elle détruit une compétence rare.

Donc le goulot n’est pas :

faire un modèle plus petit

mais :

savoir ce qu’il a perdu en devenant plus petit.

Et c’est beaucoup plus dur.

Les recherches récentes vont dans ce sens : les benchmarks statiques deviennent vulnérables à la contamination et à la saturation, d’où le passage vers des évaluations plus dynamiques ; les travaux sur le reward hacking montrent aussi qu’un modèle peut optimiser le proxy sans satisfaire l’objectif réel.

Donc le vérificateur ne doit pas être un benchmark.

Il doit être un système adversarial vivant.

2. La première brique à construire : Dynamic Skill Verifier

Le vérificateur doit devenir un moteur autonome de tests.

Je l’appellerais :

DSV — Dynamic Skill Verifier

Son rôle :

trouver ce que le modèle compressé ne sait plus faire
avant que cette perte soit enterrée dans la moyenne.

Il ne doit pas seulement poser des questions.

Il doit générer des familles de problèmes.

Pour chaque compétence, il faut définir un objet formel :

SkillSpec {
  nom_de_compétence
  générateur_de_tâches
  transformations_métamorphiques
  transformations_anti-métamorphiques
  oracle_de_vérification
  ancres_exactes
  critères_de_réussite
  critères_de_régression
  coût_de_vérification
  difficulté
}

Exemple pour l’algèbre :

SkillSpec: linear_equation_solving

générateur:
  ax + b = c

transformations_métamorphiques:
  renommer x en y
  ajouter du texte inutile
  multiplier toute l’équation par k
  reformuler en problème verbal

anti-métamorphiques:
  changer le signe
  changer une constante
  introduire division par zéro
  ajouter une contrainte contradictoire

oracle:
  vérifier par substitution

ancres_exactes:
  coefficients
  variable
  signe
  domaine

régression:
  modèle compressé échoue alors que modèle plastique réussit

Exemple pour le code :

SkillSpec: function_debugging

générateur:
  fonction avec bug contrôlé

transformations_métamorphiques:
  renommer variables
  changer indentation
  ajouter commentaires inutiles
  changer valeurs de tests

oracle:
  exécuter tests unitaires

régression:
  patch incorrect
  patch qui passe les tests visibles mais casse des tests cachés

Exemple pour long contexte :

SkillSpec: exact_anchor_retrieval

générateur:
  long document avec détails critiques

transformations:
  paraphrases
  informations distractrices
  changement d’ordre
  insertion de contradictions locales

oracle:
  comparer réponse aux ancres exactes

régression:
  nombre, nom, date, contrainte ou variable mal recopiée

Le point essentiel :

Chaque compétence doit être évaluée par une grammaire générative, pas par une liste fixe.

Cela évite que le modèle apprenne juste à passer un test.

3. Le vérificateur doit être plus créatif que le compresseur

C’est la règle centrale.

Si le compresseur est plus créatif que le vérificateur, le modèle apprend à tricher.

Il peut :

conserver les patterns visibles,
sur-optimiser les tests,
donner des réponses plausibles,
déplacer le coût dans les activations,
utiliser plus de tokens,
cacher les pertes dans le raisonnement latent,

tout en ayant perdu une vraie compétence.

Donc il faut un autre module :

Compression Adversary

Son rôle n’est pas de juger.

Son rôle est d’attaquer.

Compression Adversary:
  cherche les prompts où
  modèle compressé échoue
  mais modèle plastique réussit
  et où l’oracle confirme la bonne réponse.

L’entraînement devient un jeu :

Compressor : réduire coût
Adversary  : exposer distorsion
Verifier   : décider
Attributor : localiser la cause
Regrowth   : réparer minimalement
Compiler   : recristalliser

Ton plan Cortex-2 avait déjà cette boucle : Compressor → Adversary → Verifier → Regrowth, avec vérification dynamique et attribution causale.

Mais il faut maintenant en faire le noyau du paradigme.

Le vérificateur n’est pas un accessoire.

Il est l’équivalent des tests de non-régression dans un compilateur.

Et Cortex-3 n’est pas un modèle.

C’est un modèle plus son système de compilation, de test et de réparation.

4. Problème suivant : le vérificateur peut devenir trop cher

Si chaque génération doit être vérifiée par un gros modèle ou par des milliers de tests, on perd le gain de vitesse.

Donc le deuxième goulot est :

Comment vérifier sans refaire tout le raisonnement ?

La solution est une vérification hiérarchique.

Ton plan proposait déjà trois niveaux : vérification locale cheap, vérification ciblée, puis vérification lourde seulement pour les tâches difficiles. Je garde cette structure, mais je la rends plus stricte.

Niveau 0 — vérification syntaxique quasi gratuite

À faire systématiquement.

format correct ?
contrat FSP respecté ?
nombre d’ancres exactes conservées ?
horizon MTP cohérent ?
pas de contradiction locale ?

Coût visé : négligeable.

Ce niveau ne prouve pas que la réponse est bonne. Il détecte les erreurs évidentes.

Niveau 1 — vérification par certificat

Le modèle ne donne pas forcément toute sa chaîne de pensée.

Il donne un certificat court.

Pour math :

certificat:
  variable = x
  opération critique = soustraction puis division
  contrainte = diviseur non nul
  vérification = substitution

Pour code :

certificat:
  spécification
  invariant principal
  cas limites
  tests minimaux

Pour long contexte :

certificat:
  ancres utilisées
  contraintes citées
  source du détail exact

Le vérificateur contrôle ce certificat, pas toute la pensée.

C’est ce que j’appellerais :

Proof-Carrying Generation

Le modèle transporte une preuve compressée avec sa réponse.

Pas une longue chaîne de pensée.

Un certificat vérifiable.

Niveau 2 — vérification par outil

Activée seulement pour les domaines où l’oracle est clair :

exécution de code,
tests unitaires,
calcul symbolique,
substitution mathématique,
comparaison exacte d’ancres,
validateurs JSON/schema,
solveurs spécialisés.

Ce niveau est très puissant, parce qu’il ne dépend pas seulement d’un autre LLM.

Niveau 3 — vérification par juge lourd

À utiliser seulement quand :

forte incertitude,
tâche critique,
contradiction détectée,
compression agressive,
nouvelle compétence,
régression suspectée.

Le vérificateur lourd ne doit pas être appelé par défaut.

Sinon il devient le vrai modèle.

5. Le troisième goulot : attribuer la perte

Détecter une régression ne suffit pas.

Il faut savoir pourquoi elle arrive.

La question difficile :

la perte vient-elle des poids ternaires ?
des activations 4-bit ?
du KV latent ?
du routeur MoE ?
de l’horizon MTP ?
du contrat FSP ?
du raisonnement latent ?
du compresseur d’entrée ?
de la perte d’incertitude ?

Sans attribution, on ne sait pas réparer.

Donc il faut construire :

Causal Regression Attributor

Il observe chaque échec et produit un diagnostic.

Pour chaque réponse, Cortex-3 doit logger :

tokens compressés
ancres exactes utilisées
experts activés
profondeur utilisée
horizon MTP choisi
contrat FSP prédit
niveau de vérification
bits d’activation alloués
blocs ternaires activés
certificat produit
loops latents effectuées

Ensuite, quand une régression apparaît :

modèle plastique réussit
modèle cristallisé précédent réussit
modèle candidat échoue

on lance des contre-expériences :

même modèle, mais KV non compressé
même modèle, mais activations 8-bit
même modèle, mais horizon MTP réduit
même modèle, mais expert alternatif
même modèle, mais plus de profondeur
même modèle, mais ancres exactes forcées

Cela permet de localiser le goulot.

Exemple :

si l’erreur disparaît avec KV exact :
  problème = mémoire compressée

si l’erreur disparaît avec horizon MTP=1 :
  problème = génération trop spéculative

si l’erreur disparaît avec activation 8-bit dans bloc 17 :
  problème = précision d’activation

si l’erreur disparaît avec expert code_2 :
  problème = routage MoE

si l’erreur disparaît avec certificat enrichi :
  problème = raisonnement latent insuffisamment vérifiable

Le but n’est pas seulement de savoir que le modèle échoue.

Le but est de savoir quel mécanisme de compression l’a cassé.

6. Le quatrième goulot : réparer sans regrossir

Une fois qu’on a trouvé une perte, le réflexe naïf serait :

ajouter des paramètres
augmenter la précision
utiliser plus de couches
vérifier plus souvent

Mais ça détruit le projet.

Donc il faut une réparation minimale.

Minimal Regrowth Planner

Le regrowth ne doit pas être :

réentraîner plus

mais :

acheter la plus petite correction qui récupère la compétence.

Actions possibles :

réactiver un bloc ternaire zéro
changer quelques signes
augmenter un scale partagé
ajouter une ancre exacte
réduire l’horizon MTP pour une classe de tâches
forcer un expert spécialisé
augmenter temporairement une activation de 4-bit à 8-bit
ajouter un certificat plus riche
ajouter une règle de vérification locale
ajouter une micro-donnée de correction

Chaque réparation doit être évaluée par :

valeur = gain_compétence_vérifiée / coût_ajouté

Donc :

si une connexion récupère 0.1% de score mais coûte beaucoup,
elle est rejetée.

si une ancre exacte récupère une compétence long-contexte avec coût faible,
elle est acceptée.

si réduire l’horizon MTP évite des erreurs de code critiques,
on l’applique seulement pour ce domaine.

C’est important : le regrowth n’est pas seulement structurel.

Il peut être :

regrowth de poids,
regrowth de précision,
regrowth de mémoire,
regrowth de vérification,
regrowth de raisonnement,
regrowth de routage,
regrowth de données.

La réparation minimale peut être ailleurs que dans les poids.

7. Le cinquième goulot : empêcher le déplacement du coût

C’est l’un des problèmes les plus dangereux.

Un modèle peut prétendre être plus efficace parce que ses poids sont 1.58-bit, mais déplacer le coût dans :

activations plus riches,
KV cache énorme,
raisonnement plus long,
vérification plus fréquente,
plus d’experts actifs,
MTP rejeté trop souvent,
certificats trop longs,
plus de loops latents.

Ton plan Cortex-2 avait déjà introduit la notion de Total Cognitive Description Length, en comptant signes, masques, scales, routeurs, ancres, certificats et KV, pour éviter la fausse compression.

Il faut aller plus loin.

La métrique centrale doit devenir :

coût total par compétence correcte vérifiée

Pas :

latence par token

Pas :

bits par poids

Pas :

tokens par seconde

Mais :

J = coût_total / qualité_vérifiée

Avec :

coût_total =
  poids chargés
+ activations
+ KV cache
+ tokens générés
+ loops latents
+ experts activés
+ vérifications
+ mémoire exacte
+ rejet MTP
+ regrowth

Et :

qualité_vérifiée =
  accuracy
× robustesse métamorphique
× calibration
× fidélité aux ancres
× absence de régression
× cohérence du contrat

Cela évite le piège :

modèle rapide par token
mais lent par réponse correcte.

C’est capital pour les modèles de raisonnement, parce que générer plus vite ne sert à rien si le modèle doit produire deux fois plus de tokens de pensée.

8. Rendre le modèle beaucoup plus rapide

La vitesse ne doit pas venir d’un seul truc.

Elle doit venir de cinq compressions simultanées :

moins de bits par poids,
moins de mémoire déplacée,
moins de tokens lus,
moins de tokens générés,
moins de calcul par token.
8.1. Vitesse niveau poids : ternaire hardware-native

BitNet b1.58 montre que des poids ternaires {-1,0,+1} peuvent être utilisés dans un LLM natif 1-bit/1.58-bit ; le rapport b1.58 2B4T décrit un modèle 2B entraîné sur 4T tokens avec gains mémoire, énergie et latence annoncés face à des modèles full-precision comparables.

Mais Cortex-3 ne doit pas seulement utiliser des poids ternaires.

Il doit compiler ses poids en blocs directement exploitables par le hardware.

Au lieu de :

W_ij ∈ {-1,0,+1}

on impose :

W_block ∈ HardwareTernaryCodebook

Exemple :

bloc de 4 poids:
  exactement 1 ou 2 zéros
  signes packables
  scale partagé

Donc :

[+1, 0, -1, +1]

n’est pas juste une matrice.

C’est une instruction compacte.

Le modèle apprend à se représenter dans la grammaire du hardware.

C’est la différence entre :

quantization

et :

compilation vers machine ternaire.
8.2. Vitesse niveau activation : W1.58A4 adaptatif

BitNet v2 indique que les activations sont un vrai obstacle : les outliers compliquent la quantification basse précision ; la solution proposée utilise une transformation de Hadamard pour lisser les activations et permettre du 4-bit natif avec faible dégradation.

Mais je ne ferais pas seulement :

toutes les activations en 4-bit

Je ferais :

activation facile → 2-4 bit
activation normale → 4 bit
activation critique → 8 bit temporaire

Critique signifie :

négation,
nombre,
variable,
contrainte,
token rare,
code,
signe mathématique,
ancre exacte.

Donc le modèle alloue des bits aux activations comme il alloue de l’attention.

Formule :

precision_activation =
  f(salience_causale, incertitude, risque_compétence)
8.3. Vitesse niveau contexte : mémoire latente + registre exact

Le KV cache est un piège énorme.

DeepSeek-V3 combine MLA, MoE et objectif MTP pour réduire coût d’inférence et entraînement ; MLA vient précisément de l’idée de représenter l’attention via une forme latente plus compacte.

Cortex-3 doit faire :

KV récent exact
KV ancien latent
registre exact d’ancres
mémoire orientée requête

Donc :

texte long
→ résumé latent
+ ancres exactes

Mais les ancres sont obligatoires pour :

nombres,
noms,
dates,
variables,
contraintes,
citations,
identifiants,
code,
chemins de fichiers.

La mémoire ne doit pas seulement être compressée.

Elle doit être causalement compressée.

Même contexte, compression différente selon la question.

question: combien ?
→ garder nombres

question: où ?
→ garder lieux

question: pourquoi ?
→ garder relations causales

question: quel code ?
→ garder symboles exacts
8.4. Vitesse niveau sortie : MTP adaptatif

Le multi-token prediction donne un signal d’entraînement plus riche et peut accélérer l’inférence ; le papier de 2024 rapporte notamment des gains en code et jusqu’à 3× d’accélération avec 4-token prediction.

Mais Cortex-3 ne doit pas utiliser un horizon fixe.

Il doit choisir :

horizon MTP = f(confiance, domaine, risque, contrat)

Exemples :

texte banal → 8 tokens
liste prévisible → 6 tokens
explication standard → 4 tokens
code sensible → 2 tokens
math critique → 1 token
réponse incertaine → 1 token + vérification

PIPO va dans une direction proche : unir compression d’entrée et multi-token prediction côté sortie, avec une tête de confiance pour éviter un vérificateur trop coûteux.

Cortex-3 doit généraliser PIPO :

Variable-In Variable-Out

Pas seulement 2 tokens vers 1 latent.

Mais :

entrée:
  1 token → 1 latent si critique
  2 tokens → 1 latent si normal
  4-8 tokens → 1 latent si redondant

sortie:
  1 token si risqué
  2-4 tokens si normal
  8 tokens si très prévisible
8.5. Vitesse niveau profondeur : calcul dynamique

LayerSkip montre que l’early exit et le self-speculative decoding peuvent accélérer l’inférence en partageant poids et activations entre draft et vérification ; Mixture-of-Depths montre qu’un Transformer peut apprendre à allouer les FLOPs de manière non uniforme selon les tokens.

Cortex-3 doit utiliser trois chemins :

Fast Path:
  compression forte
  peu de couches
  MTP horizon élevé
  vérification niveau 0

Normal Path:
  compression moyenne
  profondeur normale
  MTP horizon 2-4
  certificat léger

Careful Path:
  compression prudente
  experts spécialisés
  loops latents
  horizon MTP faible
  vérification forte

Le modèle doit apprendre à router chaque demande vers le bon chemin.

Le vrai accélérateur n’est pas seulement le 1-bit.

C’est :

ne pas utiliser toute l’intelligence quand ce n’est pas nécessaire.
9. Rendre le modèle beaucoup plus intelligent

La compression seule ne suffit pas à rendre le modèle plus intelligent.

Elle peut même l’affaiblir si elle est mal faite.

Pour obtenir un changement de paradigme, il faut que Cortex-3 utilise la compression comme moteur d’abstraction.

Le mécanisme clé :

SlowSolve → Verify → Distill → Compile → FastSolve

C’est probablement la partie la plus importante de tout le projet.

Étape 1 — SlowSolve

Le modèle utilise un mode lent :

plus de profondeur,
plus d’experts,
raisonnement latent,
outils,
vérification forte,
recherche,
plusieurs candidats.

Il résout une tâche difficile.

Étape 2 — Verify

On vérifie fortement :

oracle,
tests,
certificat,
métamorphismes,
contre-exemples,
comparaison avec modèle plastique.
Étape 3 — Distill

On extrait :

la stratégie,
les invariants,
les sous-objectifs,
les erreurs évitées,
les ancres importantes,
le type de circuit utilisé.
Étape 4 — Compile

On transforme cette stratégie en :

circuit ternaire,
expert spécialisé,
règle de routage,
mémoire latente,
SkillSpec,
vérification cheap,
contrat FSP.
Étape 5 — FastSolve

La prochaine fois, le modèle n’a plus besoin de refaire tout le raisonnement.

Il utilise le circuit compilé.

Donc le modèle devient plus intelligent et plus rapide.

Pas seulement parce qu’il apprend.

Mais parce que :

ce qui était autrefois une délibération lente devient une compétence rapide.

C’est l’équivalent cognitif de compiler une fonction appelée souvent.

10. Auto-amélioration récursive : oui, mais pas naïve

Tu demandes une amélioration auto-récursive si possible.

Oui, mais il faut être très strict.

La self-improvement naïve est dangereuse :

le modèle génère ses propres données
il s’entraîne dessus
il renforce ses erreurs
il apprend à hacker son vérificateur
il s’éloigne du réel

Le risque de model collapse est réel : Nature décrit la dégradation de modèles entraînés récursivement sur du contenu généré par des générations précédentes.

Donc Cortex-3 ne doit pas faire :

je génère → je crois → je m’entraîne

Il doit faire :

je propose → je teste → je vérifie → je compare → j’archive → je compile seulement si meilleur.

Les travaux récents donnent des signaux utiles : SEAL explore des modèles qui génèrent leurs propres “self-edits” pour adapter leurs poids ; Darwin Gödel Machine explore des agents qui modifient leur propre code avec validation empirique ; AlphaEvolve combine LLMs, évolution et évaluateurs pour améliorer des algorithmes et même optimiser des parties d’infrastructure d’IA.

Mais pour Cortex-3, l’auto-amélioration doit être limitée par des preuves empiriques fortes.

11. Le système d’auto-amélioration : Recursive Improvement Engine

Je construirais un module :

RIE — Recursive Improvement Engine

Il ne modifie pas directement le modèle principal.

Il travaille en sandbox.

Il propose des changements dans plusieurs catégories.

Catégorie A — amélioration de données
nouveaux générateurs de tâches,
nouvelles transformations métamorphiques,
nouvelles classes d’équivalence,
nouveaux contre-exemples,
nouveaux problèmes de frontière.
Catégorie B — amélioration de vérification
nouvel oracle,
nouveau certificat,
nouvelle règle anti-hallucination,
nouveau test d’ancres,
nouveau détecteur d’incohérence.
Catégorie C — amélioration de compression
nouvelle grammaire de blocs ternaires,
nouvelle stratégie de zéros certifiés,
nouveau plan de regrowth,
nouveau scheduler d’activation bits.
Catégorie D — amélioration d’inférence
meilleur routeur fast/normal/careful,
meilleur horizon MTP,
meilleur early exit,
meilleur usage d’experts,
meilleure politique de KV latent.
Catégorie E — amélioration d’architecture
nouvel expert,
nouveau module latent,
nouvelle tête FSP,
nouvelle tête de confiance,
nouveau compresseur d’entrée.
Catégorie F — amélioration de kernels
meilleur packing ternaire,
meilleur kernel N:M,
meilleure fusion de kernels,
meilleur layout mémoire.

Le RIE propose un patch.

Puis :

1. entraînement sandbox
2. évaluation dynamique
3. comparaison contre référence
4. test anti-régression
5. mesure coût total
6. acceptation seulement si Pareto-amélioration

Acceptation :

Accepter M' si :

qualité_vérifiée(M') ≥ qualité_vérifiée(M) + seuil
OU
coût_total(M') ≤ coût_total(M) - seuil

ET aucune compétence protégée ne régresse
ET la calibration ne baisse pas
ET le vérificateur adversarial ne trouve pas de faille critique
ET la performance ne dépend pas d’un benchmark fixe

Sinon :

rejeter ou archiver comme variante intéressante.
12. Pourquoi archiver les variantes est crucial

Ne garder que le meilleur modèle immédiat est dangereux.

Cela mène à des optima locaux.

Il faut un archive évolutionnaire :

modèles plus rapides,
modèles plus robustes,
modèles meilleurs en code,
modèles meilleurs en maths,
modèles plus petits,
modèles avec meilleur routage,
modèles avec meilleur verifier,
modèles avec meilleure mémoire.

Puis on peut faire du croisement :

prendre le routeur du modèle A
prendre le compresseur KV du modèle B
prendre l’expert code du modèle C
prendre le scheduler MTP du modèle D

Ce n’est plus un simple entraînement.

C’est une écologie de modèles candidats.

Mais chaque candidat doit passer par :

vérification,
non-régression,
coût total,
tests dynamiques,
anti-collapse,
anti-reward-hacking.
13. Le vrai moteur d’intelligence : Frontier Skill Discovery

Pour devenir beaucoup plus intelligent, Cortex-3 doit chercher sa frontière.

Pas seulement apprendre des données passives.

Il doit demander :

quelles tâches sont juste au-dessus de mon niveau actuel ?

Je construirais :

Frontier Skill Discovery

Processus :

1. Le Skill Ledger identifie une zone fragile.
2. L’adversaire génère des problèmes proches de la frontière.
3. Le modèle tente en mode lent.
4. Si échec, un solver externe ou professeur aide.
5. La solution est vérifiée.
6. Les invariants sont extraits.
7. La compétence est distillée.
8. Elle est compilée dans un circuit rapide.

Exemple :

le modèle échoue sur équations avec contraintes implicites
→ générer 10 000 variantes contrôlées
→ résoudre avec oracle symbolique
→ extraire stratégie
→ distiller en expert algèbre
→ compiler en ternaire
→ tester métamorphiquement

Pour le code :

le modèle échoue sur bugs d’état mutable
→ générer programmes avec mutations contrôlées
→ tests unitaires cachés
→ extraire invariant
→ créer expert debugging_state
→ compiler

Pour long contexte :

le modèle perd les détails exacts après compression
→ générer documents longs avec ancres
→ entraîner registre exact
→ apprendre compression orientée requête

Donc Cortex-3 apprend en ciblant ses faiblesses.

C’est très différent de :

préentraîner sur masse de texte.

15. Le prochain goulot après le vérificateur : l’oracle

Même un vérificateur dynamique a besoin d’oracles.

Certains domaines sont faciles :

math calculable,
code exécutable,
formats,
tests unitaires,
retrieval exact.

D’autres sont difficiles :

qualité d’un raisonnement ouvert,
factualité complexe,
jugement nuancé,
créativité,
planification longue.

Donc le prochain goulot après le vérificateur est :

Comment obtenir une vérité vérifiable dans les domaines non formels ?

Solution :

transformer autant de tâches que possible en sous-composants vérifiables.

Exemple : réponse longue.

Au lieu de vérifier globalement :

cette réponse est-elle bonne ?

on vérifie :

a-t-elle répondu à la question ?
a-t-elle respecté les contraintes ?
a-t-elle conservé les ancres ?
a-t-elle évité contradiction ?
a-t-elle signalé l’incertitude ?
a-t-elle utilisé les bonnes sources ?
a-t-elle un plan cohérent ?

Donc on ne demande pas au juge :

est-ce intelligent ?

On lui demande :

quelles obligations ont été satisfaites ?

C’est beaucoup plus robuste.

16. Le goulot suivant : intelligence générale vs compétences locales

Un risque : Cortex-3 pourrait devenir excellent sur des SkillSpecs bien définis, mais moins bon sur les situations ouvertes.

Compétences atomiques

suivi d’entité,
debugging,
récupération d’ancre,

Compétences compositionnelles
combiner plusieurs compétences,
planifier,
choisir une stratégie,
savoir quand vérifier,
savoir quand demander clarification,
savoir quand utiliser un outil.

Le Skill Ledger doit donc avoir deux niveaux :

Skill Ledger atomique
Meta-Skill Ledger

Meta-compétences à entraîner :

détecter difficulté,
choisir un chemin d’inférence,
choisir un expert,
choisir le bon horizon MTP,
choisir quoi mémoriser exactement,
choisir quoi compresser,
choisir quand douter,
choisir quand vérifier.

C’est là que le modèle devient plus intelligent.

Pas seulement parce qu’il sait plus de choses.

Mais parce qu’il sait mieux gérer son propre calcul.
